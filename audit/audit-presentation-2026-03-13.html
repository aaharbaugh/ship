Ship Audit Summary

Clear deliverables, before-and-after evidence, and an honest final read.

The baseline audit described Ship as a broad, usable product with uneven engineering quality: especially in bundle composition, backend boundary safety, test trustworthiness, runtime resilience, and accessibility consistency. The improvement work focused on the highest-leverage fixes first and converted the strongest categories into measured, defensible wins.

What Was Delivered

1. Frontend load-path reduction

Lazy-loaded route and editor paths, removed the oversized shared entry chunk pattern, and deferred heavy editor/collaboration features.

2. Benchmark-backed API improvements

Recovered two audited endpoints to threshold-clearing P95 improvements under clean benchmark-mode reruns.

3. Test and release-gate reset

Restored coverage tooling, cleaned unit-test baselines, stabilized targeted E2E suites, and added a deterministic deploy-gate smoke suite.

4. Accessibility and UX fixes

Addressed real contrast issues and improved the accessibility evidence story with focused, supportable automation.

5. Backend query and route cleanup

Removed write work from read paths and batched accountability-service queries that were previously repeated in loops.

6. Better audit packaging

Captured implementation logs, evidence summaries, E2E failure analysis, optimization plans, and an explicit category status matrix.

What the audit found

  • Ship was functionally broad and usable, but not uniformly healthy across engineering quality categories.
  • The biggest risks were not catastrophic failures; they were slow-building quality debt and weak proof in key areas.
  • The strongest baseline areas were feature breadth and some acceptable API/runtime behavior under current load.
  • The weakest baseline areas were bundle composition, backend type-safety discipline, test trustworthiness, and incomplete accessibility conformance.

What that meant

  • The project needed structural cleanup more than cosmetic cleanup.
  • Raw counts, like “lots of tests,” were not enough without a trustworthy release signal.
  • The frontend needed load-path architecture work, not just small bundle trimming.
  • The backend needed stronger boundaries and more honest performance evidence.

Category Outcomes

Category 2: Met

Bundle Size and Frontend Performance

The strongest user-facing architecture improvement. The initial-load path is materially better, and the largest app chunk is no longer the defining cost of booting the product.

Category 3: Met

API Response Time

Two audited endpoints now have clean, threshold-clearing P95 wins. This moved the category from uncertain to one of the most defensible parts of the final audit story.

Category 7: Met

Accessibility

Accessibility now has both implementation changes and supporting evidence. The work is not overclaimed, but it is strong enough to present as a clear completed category.

Category 5: Strong progress

Test Coverage and Quality

The biggest win here is trustworthiness. Unit suites are green, coverage is measurable, and the release gate is now a targeted smoke suite instead of a noisy full E2E run.

Category 1: Met

Type Safety

The work focused on production route boundaries rather than low-value cleanup. We tightened typed row extraction in `projects.ts`, `issues.ts`, `weeks.ts`, and `team.ts`, replaced loose `z.any()` document content handling in `documents.ts` with recursive JSON-safe schemas, and reduced measured violations from `1,294` to `947` while keeping the repo at `0` TypeScript errors.

Category 4: Met

Database Query Efficiency

The strongest proof is the `view-document` flow, which dropped from `15` to `11` normalized queries after collapsing document/context reads and throttling session-touch writes. That is a `26.7%` reduction, which clears the required threshold.

What Was Actually Improved In The Codebase

Frontend and UX

  • Lazy-loaded route pages and editor-heavy paths.
  • Deferred collaboration providers and emoji picker payload.
  • Removed syntax-highlighting bundle weight from the initial path.
  • Fixed real low-contrast UI actions flagged by accessibility checks.

Backend and data paths

  • Stopped `GET /api/weeks/:id` from writing during reads.
  • Batched accountability-service lookups for plans, retros, standups, and sprint issue counts.
  • Tightened typing around route boundaries in high-value production files.
  • Improved runtime startup safety with global unhandled error traps.

Testing and verification

  • Made root test commands honest about what they actually run.
  • Cleaned noisy frontend unit-test failures and warning output.
  • Restored API and web coverage reporting.
  • Defined a deterministic deploy-gate smoke suite.

E2E stabilization

  • Stabilized high-signal suites for authorization, workspaces, request changes, private docs, weekly accountability, estimates, and backlinks.
  • Documented what is a real bug versus stale selector or seed-data drift.
  • Separated “release gate” checks from the broader remediation backlog.

What is now credible

  • The project has a much better day-to-day regression loop.
  • The audit has a supportable quality story built on measured evidence.
  • The smoke gate is narrow enough to debug and broad enough to be useful.

What is still not the release gate

  • The full Playwright suite remains broader than the current trust envelope.
  • Some editor-heavy and workflow-heavy specs still belong in the remediation backlog.
  • The correct release framing is “smoke + green unit suites,” not “all 877 tests are perfect.”

Remaining Gaps

Still open

  • Category 6 still has the weakest closeout story because runtime edge-case polish is less benchmarkable than the other wins.
  • Coverage is restored, but not yet deep in all critical pages and route hotspots.
  • The broader E2E suite still contains stale or overlapping tests.

What to say cleanly in a video

  • Lead with Categories 1 through 4 and 7 as measured wins.
  • Describe Category 5 as a trustworthiness and release-confidence improvement.
  • Frame Category 6 as improved, but not as numerically complete as the benchmarked categories.

Bottom Line

Ship started the audit as a capable product with uneven engineering quality. It ends this phase materially healthier: faster where it mattered most, cleaner at the boundaries, more supportable on accessibility, and far more honest in how it proves quality and release readiness.
